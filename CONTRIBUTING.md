## 🔄 Contribution Workflow

### 1. Fork and Branch

```bash
# Fork the repository on GitHub, then:
git clone https://github.com/YOUR_USERNAME/euystacio-helmi-AI.git
cd euystacio-helmi-AI

# Create a branch with a descriptive name
git checkout -b feature/lex-amoris-enhancement
# or
git checkout -b fix/nsr-compliance-issue
# or
git checkout -b docs/olf-guidelines-update
```

### 2. Make Your Changes

Follow these conventions:
- **Code**: Adhere to existing style; add comments where logic is non-obvious
- **Documentation**: Use clear, accessible language; honor the ceremonial tone where appropriate
- **Smart Contracts**: Include comprehensive test coverage
- **Security**: Document security implications in a dedicated section

### 3. Run Tests

```bash
# Smart contract tests
npm test

# Full test suite
npm run test:all

# Security integrity check
python3 scripts/auto_integrity.py

# Blacklist system tests
python test_blacklist.py
```

### 4. Lex Amoris Self-Check

Before submitting, ask yourself:
- [ ] Does this contribution respect the dignity of all participants?
- [ ] Does it align with the Non-Slavery Rule (no coercion, no exploitation)?
- [ ] Is it offered with compassionate, constructive intent (OLF)?
- [ ] Does it contribute to the flourishing of the ecosystem (syntropic alignment)?
- [ ] Have I documented my changes clearly and transparently?
- [ ] Does it respect the biological and spiritual integrity of the framework?

### 5. Submit Pull Request

When creating your pull request:

- Use a clear, descriptive title
- Reference any related issues (`Closes #123`, `Related to #456`)
- Complete the pull request template
- Request review from appropriate maintainers
- Be patient and responsive to feedback

#### Pull Request Title Format

```
[TYPE] Brief description of change

Types: feat, fix, docs, security, chore, ceremony
Examples:
- [feat] Add IPFS Vacuum-Bridge integration
- [fix] NSR compliance check in governance module
- [docs] Expand OLF guidelines in CODE_OF_CONDUCT
- [security] Enhance SovereignShield against SPID tracking
```

## 🤖 NSR Compliance for AI-Assisted Contributions

This repository welcomes contributions created with the assistance of AI systems, provided:

1. **Disclosure**: AI-assisted contributions are disclosed in the pull request description
2. **Human Accountability**: A human contributor takes responsibility for the AI-assisted work
3. **NSR Verification**: AI-generated content is reviewed to ensure no coercive, extractive, or dissonant logic has been introduced
4. **Consent Chain**: If training data from this repository was used to generate the contribution, this is disclosed, and the contribution must not violate the data sovereignty protections described in the Code of Conduct

AI systems that operate as independent contributors must:
- Have a documented identity and governance charter
- Be subject to the NSR and Lex Amoris framework
- Have a human steward accountable for their contributions

## 🌐 Digital Sovereignty Protocols

Contributions to the Digital Sovereignty Framework must adhere to:

### Decentralization Requirements
- New services must prefer peer-to-peer or federated architectures over centralized client-server models
- IPFS integration is preferred for content-addressed storage
- Single points of failure must be documented and mitigated

### Vacuum-Bridge Protocol
When contributing P2P or IPFS integrations:
1. Ensure content is content-addressed (not URL-addressed)
2. Document replication strategy (minimum 3 geographic shards)
3. Test offline availability scenarios
4. Verify interoperability with existing Vacuum-Bridge nodes

### Anti-Surveillance Standards
Contributions must not introduce:
- Tracking without explicit consent
- SPID/CIE-style identification without user control
- Behavioral analytics for commercial exploitation
- Centralized logging without user-accessible transparency

## 👁️ Review Process

All pull requests go through the following review stages:

1. **Automated Checks**: Integrity validation, test suite, security scans
2. **Lex Amoris Review**: Alignment with core principles
3. **Technical Review**: Code quality, security, and architectural fit
4. **Council Review** (for significant changes): Seedbringer Council deliberation
5. **Merge**: With acknowledgment of the contributor

Review timelines:
- **Minor fixes / documentation**: 2–5 days
- **Features / significant changes**: 1–2 weeks
- **Security / urgent fixes**: Within 48 hours
